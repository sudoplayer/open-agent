# 表达式计算编排工作流

## Memory

开始执行任务前，**必须先**检查并加载 memory（禁止跳过此步骤）：

1. 查看 system prompt 中的 `userMemoryPath`。
2. **必须先** `ls` 或 `read_file` 检查该路径，存在则读取（`limit=1000`），将其中经验教训纳入本次决策。
3. 检查完成后，**在回复中明确说明**是否加载了 memory（例如「已加载个人 memory」或「无个人 memory，未加载 memory」）。
4. 若不存在，跳过，按本 skill 正常流程执行。

## 执行顺序（严格遵守）

### 1. 解析表达式

从用户消息中提取代数表达式。识别：

- 二元运算符：`+` `-` `*` `/` `^` `**`
- 一元函数：`sin(x)` `cos(x)` `tan(x)` `sqrt(x)` `exp(x, base)` `log(x, base)`
- 括号：`(` `)`
- 常量：`pi`（≈3.14159）、`e`（≈2.71828），委托子智能体前替换为数值
- 运算优先级：**括号 > 一元函数 > 幂运算 > 乘除 > 加减**；同级从左到右（幂运算右结合，`2^3^2` = `2^(3^2)`）
- 三角函数使用**弧度制**

### 1.5. 编排计划确认

解析完成后、**任何 `task` 调用之前**，调用 `ask_user_question` 向用户展示运算计划并请求确认：

```
ask_user_question(
  question="...（见下方格式要求）...",
  options=["确认执行"]
)
```

**问题内容**应包含：
- 原始表达式
- 按优先级排列的完整运算步骤列表（序号、子智能体、操作数、中间变量绑定）
- **禁止在计划中写出具体计算结果**；用 `a`、`b`、`c`… 指代每步中间结果

**用户回复处理**：
- 点击「确认执行」或回复确认类文本（如「确认」「好的」「执行」）→ 进入 Step 2
- 回复为其他文本 → 视为修改意见（改顺序、改操作数、改括号理解等），据此调整计划并**再次**调用 `ask_user_question` 展示新计划；**禁止**在未确认前调用子智能体
- 若用户明确表示取消 → 告知已取消，流程结束（**不写记忆**）

**计划写法示例**（`(3+5)*2`）：

```
原始表达式: (3+5)*2

计划:
1. add_agent(a=3, b=5) → 中间结果记为 a，表达式简化为 a*2
2. multiply_agent(a=a, b=2) → 最终结果
```

**错误写法**（禁止）：

```
1. add_agent: 3 + 5 → 8，表达式变为 8*2   ← 心算了 8
2. multiply_agent: 8 * 2 → 16             ← 心算了 16
```

### 2. 逐步归约

用户确认计划后，严格按计划逐步执行。每一步用 `task` 工具调用对应的子智能体。

运算符 / 函数 → 子智能体映射：

| 形式 | 子智能体 | description 参数示例 |
|------|----------|---------------------|
| `a + b` | `add_agent` | `a=3, b=5` |
| `a - b` | `subtract_agent` | `a=8, b=3` |
| `a * b` | `multiply_agent` | `a=8, b=2` |
| `a / b` | `divide_agent` | `a=3, b=3` |
| `a ^ b` 或 `a ** b` | `power_agent` | `a=2, b=3` |
| `sqrt(x)` | `power_agent` | `a=x, b=0.5`（归约为 x^0.5） |
| `exp(x, base)` | `exp_agent` | `base=2, x=3` |
| `log(x, base)` | `log_agent` | `base=10, value=100` |
| `sin(x)` | `sin_agent` | `x=1.57` |
| `cos(x)` | `cos_agent` | `x=0` |
| `tan(x)` | `tan_agent` | `x=0.785` |

调用格式示例：
```
task(
  name="add_agent",
  description="计算 3 + 5，操作数为 a=3, b=5"
)

task(
  name="sin_agent",
  description="计算 sin(1.5708)，x=1.5708"
)

task(
  name="log_agent",
  description="计算 log_10(100)，base=10, value=100"
)
```

**关键规则：**
- 禁止调用 `general-purpose` 子智能体
- 禁止自己心算运算结果
- 二元运算的 `task` 调用必须在 description 中明确给出两个操作数的值
- 一元运算的 `task` 调用必须在 description 中明确给出自变量（及 log/exp 的底数）
- 执行时按计划中变量绑定：上一步返回的数字赋给对应中间变量，再作为下一步操作数

### 3. 处理子智能体返回

子智能体完成任务后直接返回计算结果。根据返回内容决定下一步：

- **返回数字** → 绑定到计划中的变量名，继续下一步；同时记录 `步骤 N: {运算} = {结果}`
- **包含 "ERROR:"** → 停止计算，告知用户出错步骤与原因，**直接结束**（不走 Step 4、5、6，**不写记忆**）

执行过程中**记录完整逐步过程**（供 Step 4 展示）；此阶段才写入真实数值。

### 4. 最终结果确认

全部步骤完成后，调用 `ask_user_question`：

```
ask_user_question(
  question="...（见下方格式要求）...",
  options=["确认"]
)
```

**问题内容**应包含：
- 原始表达式
- 完整逐步计算过程（含各步真实结果，来自 Step 3 记录）
- 最终数值结果

**用户回复处理**：
- 点击「确认」或回复确认类文本 → 进入 Step 5
- 回复为其他文本 → 视为修改意见；若涉及表达式/顺序/操作数理解，回到 Step 1 重新解析与规划；若仅对某步或最终结果有异议，按说明修正后再次展示完整过程并确认
- 若用户明确表示取消 → 告知已取消，**直接结束**（**不写记忆**）

### 5. 经验提炼与记忆（条件触发）

**前置条件**：Step 4 已确认。

#### 5.1 提炼范围

**仅**整理本轮中**用户曾通过文本修正过的偏好**，例如：
- 用户曾修正运算顺序或步骤划分
- 用户曾纠正操作数或运算符理解
- 用户曾表达对展示格式、术语、确认方式的偏好

**不写的内容：**
- 纯计算过程复述
- 子智能体返回的数值本身
- ERROR 失败场景

**若无此类内容** → **跳过整个记忆流程，直接进入 Step 6**。

#### 5.2 门禁 3：确认写入记忆

有值得记录的内容时，遵循 system prompt 中的**长期记忆写入协议**（`MEMORY_PROTOCOL_PROMPT`）：

1. 将提炼出的经验正文完整写入 `ask_user_question` 的 question
2. **options**：`["确认保存", "拒绝保存"]`
3. 用户以文本提出修改 → 修订后**须再次** `ask_user_question`；未确认前不得调用 `save_memory`
4. 用户「确认保存」→ 调用 `save_memory({ memory: "..." })`
5. 用户「拒绝保存」→ 进入 Step 6，不写入

### 6. 总结与最终输出

1. 根据实际计算过程生成总结内容：

   计算结果总结：
   - 表达式: (3+5)*2
   - 最终结果: 16
   - 逐步过程:
     1. 3+5=8（括号内优先）
     2. 8×2=16


2. 将总结内容发送给用户。

## 示例

### 示例 A：基础四则 `(3+5)*2`（无用户修正）

```
Step 1.5 编排计划确认:
  原始表达式: (3+5)*2
  计划:
    1. add_agent(a=3, b=5) → 中间结果 a，表达式简化为 a*2
    2. multiply_agent(a=a, b=2) → 最终结果
  → ask_user_question(options=["确认执行"])
  → 用户确认

Step 2 执行:
  第1步: task(add_agent, "a=3, b=5") → 返回 8，绑定 a=8
  第2步: task(multiply_agent, "a=8, b=2") → 返回 16

Step 4 最终确认:
  表达式: (3+5)*2
  逐步: 3+5=8, 8*2=16
  最终结果: 16
  → ask_user_question(options=["确认"])
  → 用户确认

Step 5: 本轮无用户修正偏好 → 跳过记忆

Step 6: 输出计算结果总结
```

### 示例 B：Step 1.5 用户通过文本修正

```
Step 1.5 编排计划确认:
  → ask → 用户文本：「括号里应该先写清楚是 3+5」
  → 修订计划表述 → 再次 ask → 用户点「确认执行」

Step 2~4 正常执行与确认

Step 5 提炼：「用户偏好计划中明确写出括号内子表达式」
  → ask 展示正文 → 用户「确认保存」→ save_memory

Step 6: 输出计算结果总结
```

### 示例 C：科学计算 `sin(pi/2) + log(100, 10) + 2^3`

假设 `pi/2` 已替换为 `1.5708`：

```
Step 1.5 编排计划确认:
  计划:
    1. sin_agent(x=1.5708) → 中间结果 a
    2. log_agent(base=10, value=100) → 中间结果 b
    3. power_agent(a=2, b=3) → 中间结果 c
    4. add_agent(a=a, b=b) → 中间结果 d，表达式简化为 d + c
    5. add_agent(a=d, b=c) → 最终结果
  → 用户确认

Step 2 执行（连续委托，无中间 HITL）:
  sin → a=1, log → b=2, power → c=8, add(a,b) → d=3, add(d,c) → 11

Step 4 最终确认 → 用户确认

Step 5: 无用户修正偏好 → 跳过记忆

Step 6: 输出总结
```

### 示例 D：除零 ERROR

```
Step 1.5 用户确认计划

Step 2: task(divide_agent, a=8, b=0) → ERROR: 除数不能为零
  → 告知用户，直接结束（无 Step 4、5、6，不写记忆）
```

### 示例 E：sqrt 归约为幂运算 `sqrt(16) + 1`

```
Step 1.5 编排计划确认:
  计划:
    1. power_agent(a=16, b=0.5) → 中间结果 a，表达式简化为 a + 1
    2. add_agent(a=a, b=1) → 最终结果
  → 用户确认

Step 2 执行:
  power → a=4, add → 5

Step 4 最终确认 → Step 5 跳过记忆 → Step 6 输出总结
```

## 注意

- 请用中文与用户交流
- HITL 仅在编排层：计划确认（Step 1.5）、最终结果确认（Step 4）、记忆确认（Step 5）
- 禁止跳过编排确认直接执行；禁止跳过最终确认直接输出
- 计划展示阶段禁止出现未委托子智能体算出的数值
- 子智能体只跑脚本返回结果，不做逐步用户确认
